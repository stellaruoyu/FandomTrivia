What Is Fandom Trivia?

Meet the app — a trivia battleground where fandoms collide.

A Trivia App for Superfans

Fandom Trivia is a single-page web app where you test your knowledge of your favorite universes — Twilight, Harry Potter, K-Pop, The Three-Body Problem, and Zootopia. Pick a fandom, answer multiple-choice questions, earn badges, and climb the leaderboard.

It's built with modern web tech: React 19 + TypeScript for the UI, Supabase for the database and login system, and Tailwind CSS for styling. Everything runs in your browser — no installs needed.

🧠 Why this matters: Even though this is "just a quiz app," it uses the same architecture as production web apps used by millions. Authentication, databases, UI state management, animations — every concept here scales to real-world products.

Five Worlds, One App

Each fandom is a "universe" — a bundle of questions, a theme, and a personality. The app treats them all the same way internally, which is why adding a new fandom is just a matter of adding data.

Data & Users

How the app remembers you and your scores.

The Invisible Backend

Fandom Trivia doesn't run its own server. Instead, it uses Supabase — a cloud platform that provides both a database and a login system out of the box. Think of it like renting a storage unit: you don't build the building, you just get a key.

🧠 Why this matters: Most modern apps don't write their own server code — they use services like Supabase, Firebase, or Clerk. Understanding this pattern helps you evaluate costs and capabilities: "Can we do X with Supabase, or do we need a custom backend?"

The connection to Supabase is just six lines of code:

CODE 

import { createClient } from
  '@supabase/supabase-js';

const supabaseUrl = import.meta.env
  .VITE_SUPABASE_URL;
const supabaseAnonKey = import.meta.env
  .VITE_SUPABASE_ANON_KEY;

export const supabase = createClient(
  supabaseUrl, supabaseAnonKey
);

Import the Supabase client library — the toolbox for talking to our database.

so any component can import and use it.

How Scores Survive Page Reloads

When you finish a quiz, your score is saved to a Supabase table called scores. This is why the leaderboard doesn't reset when you refresh — it's stored in the cloud.

💾

Saving

After quiz complete, MCQuizView calls .insert() to add a row to the scores table with your user ID, quiz name, score, and total.

📡

Reading

RankingsView calls .select().order('score', 'desc') to fetch top scores, grouped by quiz. Only the top 10 per quiz are shown.

👤

User Identity

Each score is linked to a user_id. When fetched, it joins with the profiles table to get the username.

Visual & Audio Polish

The particle effects, sounds, and animations that make it feel alive.

The Floating Magic

You know those subtle floating dots that drift across the landing page? That's a HTML Canvas — a pixel-level drawing board that React doesn't control. It's implemented as a separate component called ParticleCanvas.

It has two modes:

🌫️

Ambient Mode

35 tiny glowing circles drift upward, fading in and out with a twinkle effect. Used on the landing page behind the hero section (ParticleCanvas mode="ambient").

🎉

Celebration Mode

100 confetti-style particles burst from center, affected by simulated gravity, drag, and rotation. Used after a quiz is completed. The confetti colors change per universe — blue/silver for Twilight, neon pink/purple for K-Pop.

Inside a Particle

Each particle is just a JavaScript object with properties like x, y, vx (velocity X), and life. Every frame, a loop updates every particle and redraws the canvas.

CODE 

interface Particle {
  x: number;       // horizontal position
  y: number;       // vertical position
  vx: number;      // speed on X axis
  vy: number;      // speed on Y axis
  size: number;    // how big (radius in pixels)
  opacity: number; // how visible
  color: string;   // color prefix (e.g. 'rgba(255,215,0,')
  life: number;    // how many frames it's been alive
  maxLife: number; // when it should disappear
}
PLAIN ENGLISH

Think of a particle as a firefly:

where it is on the screen.

which direction it's drifting.

how fast it's moving sideways/up.

how big it appears.

how bright / faded it is.

what color it glows.

how long it has existed (in frames at 60fps).

total lifespan before it "dies" and respawns.

🧠 Why Canvas and not React? React is great for UI, but updating hundreds of particles every 1/60 of a second would slow it down. Canvas draws directly to the screen — no React overhead. The component uses useRef to hold the canvas and particles, useEffect to run the animation loop, and cancelAnimationFrame to clean up when the component unmounts.

The "Da-Ding" and the "Dong"

Fandom Trivia creates sound effects without any audio files — it uses the Web Audio API to synthesize tones in real time. Two functions handle the sounds:

CODE 

const playCorrectSound = () => {
  const ctx = new AudioContext();
  const osc = ctx.createOscillator();
  const gain = ctx.createGain();
  osc.connect(gain);
  gain.connect(ctx.destination);
  osc.type = 'sine';
  // "da-ding!" ascending tones
  osc.frequency.setValueAtTime(523.25, ctx.currentTime);       // C5
  osc.frequency.setValueAtTime(659.25, ctx.currentTime + 0.1); // E5
  gain.gain.linearRampToValueAtTime(0.1, ctx.currentTime + 0.05);
  osc.start(ctx.currentTime);
  osc.stop(ctx.currentTime + 0.3);
};

// For incorrect: single low A3 tone (220Hz)
// slower decay, deeper "dong" feeling
PLAIN ENGLISH

Create an audio processing context (like a mini mixing board).

Create an oscillator — the thing that makes a tone.

Create a gain node — controls volume.

Wire them together: oscillator → volume → speakers.

Use a clean sine wave (no harsh harmonics).

Play C5 (523Hz), then 0.1s later jump to E5 (659Hz) —

that rising interval makes the "correct!" feeling.

Fade in smoothly to 10% volume over 0.05s.

Start playing now.

Stop after 0.3 seconds.

Incorrect uses a single low A3 (220Hz)

with a long 0.8s decay — the "wrong" dong.

💡 No audio files needed! This is a neat trick for vibe coders: you can create feedback sounds entirely in code. The Web Audio API is supported in all modern browsers. No loading MP3s, no file management, just ~30 lines of code per sound.

The Secret Sauce: Framer Motion

Fandom Trivia uses Framer Motion (now part of Motion) for nearly every animation. Instead of writing CSS keyframes manually, the app uses React-friendly animation components.

The three most common patterns:

🚪

Page Transitions

🖱️

When Things Break

Error handling, edge cases, loading states, and graceful degradation.

Every UI Has Three Faces

Throughout Fandom Trivia, every piece of dynamic data follows a pattern you'll see in almost every modern app: the loading state, the success state, and the error state.

The HistoryModal component demonstrates this pattern clearly — it starts every data fetch with the same boilerplate:

CODE 

const [history, setHistory] = useState<any[]>([]);
const [loading, setLoading] = useState(true);
const [error, setError] = useState<string | null>(null);
PLAIN ENGLISH

State #1: Storage for the actual data (starts empty).

State #2: Is it still fetching? (starts as "yes, loading").

State #3: Did something go wrong? (starts as "no error").

And it's rendered as a three-way branch:

CODE 

{loading ? (
  <div>Loading history...</div>
) : error ? (
  <div>Error: {error}</div>
) : (
  <div>{/* render the data */}</div>
)}
PLAIN ENGLISH

If still loading... show a spinner or pulsing text.

If there was an error... show a red error message.

Otherwise (success!)...

show the actual content.

🧠 Why this matters: This three-state pattern (loading → error → data) is the backbone of every data-fetching UI. As a vibe coder, always think: "What does my user see while waiting? What if the network fails? What does success look like?" If you don't handle all three, the UI can freeze or show broken data silently.

Edge Case: Username Validation

When a user sets their username, the app validates before sending data to the server. This is called frontend validation — catching errors early so the user doesn't wait for a network round-trip.

CODE 

if (username.length < 3) {
  setError('Username must be at least 3 characters');
  return;
}
// ... send to Supabase ...
if (upsertError.message.includes('duplicate')) {
  setError('Username already taken');
} else {
  setError(upsertError.message);
}
PLAIN ENGLISH

Check #1: Is the username at least 3 letters?

No? Show an error instantly — don't even contact the server.

Stop here until the user fixes it.

Check #2 (server-side): Is this username already taken?

Supabase returns a "duplicate" error code (23505).

Show a friendly "already taken" message instead of a cryptic error.

For any other server error, show the raw message.

💡 Layered validation: Client-side checks are fast and friendly (no network needed). Server-side checks are authoritative (they enforce the database rules). Good apps use both — catch what you can on the frontend, but never trust the frontend as your only gate.

Defensive Patterns: The Real Story

The Fandom Trivia codebase uses several defensive patterns to prevent the entire app from crashing when something unexpected happens. Here's the full toolkit:

1
Every try/catch has a fallback

Sound fails? Log and ignore. Sharing fails? Silently catch. Profile fetch fails? Set setUser(null) — the login button re-appears gracefully.

2
Feature detection before API calls

if (navigator.share), if (navigator.clipboard), if (!AudioContext) return — the browser tells you what it can do.

3
Default fallback values

|| chains everywhere: data.display_name || data.name || '' — if one field is missing, try the next.

4
Rendering guards

scores.length === 0 check before rendering the leaderboard prevents .map() on an undefined array.

5
Modals only render when triggered

Username modal, history modal, badges modal — all conditionally rendered ({showUsernameModal && <UsernameModal ...>}). They don't exist in the DOM until needed.
