Visual & Audio Polish

The particle effects, sounds, and animations that make it feel alive.

The Floating Magic

You know those subtle floating dots that drift across the landing page? That's a HTML Canvas — a pixel-level drawing board that React doesn't control. It's implemented as a separate component called ParticleCanvas.

It has two modes:

🌫️

Ambient Mode

35 tiny glowing circles drift upward, fading in and out with a twinkle effect. Used on the landing page behind the hero section (ParticleCanvas mode="ambient").

🎉

Celebration Mode

100 confetti-style particles burst from center, affected by simulated gravity, drag, and rotation. Used after a quiz is completed. The confetti colors change per universe — blue/silver for Twilight, neon pink/purple for K-Pop.

Inside a Particle

Each particle is just a JavaScript object with properties like x, y, vx (velocity X), and life. Every frame, a loop updates every particle and redraws the canvas.

CODE 

interface Particle {
  x: number;       // horizontal position
  y: number;       // vertical position
  vx: number;      // speed on X axis
  vy: number;      // speed on Y axis
  size: number;    // how big (radius in pixels)
  opacity: number; // how visible
  color: string;   // color prefix (e.g. 'rgba(255,215,0,')
  life: number;    // how many frames it's been alive
  maxLife: number; // when it should disappear
}
PLAIN ENGLISH

Think of a particle as a firefly:

where it is on the screen.

which direction it's drifting.

how fast it's moving sideways/up.

how big it appears.

how bright / faded it is.

what color it glows.

how long it has existed (in frames at 60fps).

total lifespan before it "dies" and respawns.

🧠 Why Canvas and not React? React is great for UI, but updating hundreds of particles every 1/60 of a second would slow it down. Canvas draws directly to the screen — no React overhead. The component uses useRef to hold the canvas and particles, useEffect to run the animation loop, and cancelAnimationFrame to clean up when the component unmounts.

The "Da-Ding" and the "Dong"

Fandom Trivia creates sound effects without any audio files — it uses the Web Audio API to synthesize tones in real time. Two functions handle the sounds:

CODE 

const playCorrectSound = () => {
  const ctx = new AudioContext();
  const osc = ctx.createOscillator();
  const gain = ctx.createGain();
  osc.connect(gain);
  gain.connect(ctx.destination);
  osc.type = 'sine';
  // "da-ding!" ascending tones
  osc.frequency.setValueAtTime(523.25, ctx.currentTime);       // C5
  osc.frequency.setValueAtTime(659.25, ctx.currentTime + 0.1); // E5
  gain.gain.linearRampToValueAtTime(0.1, ctx.currentTime + 0.05);
  osc.start(ctx.currentTime);
  osc.stop(ctx.currentTime + 0.3);
};

// For incorrect: single low A3 tone (220Hz)
// slower decay, deeper "dong" feeling
PLAIN ENGLISH

Create an audio processing context (like a mini mixing board).

Create an oscillator — the thing that makes a tone.

Create a gain node — controls volume.

Wire them together: oscillator → volume → speakers.

Use a clean sine wave (no harsh harmonics).

Play C5 (523Hz), then 0.1s later jump to E5 (659Hz) —

that rising interval makes the "correct!" feeling.

Fade in smoothly to 10% volume over 0.05s.

Start playing now.

Stop after 0.3 seconds.

Incorrect uses a single low A3 (220Hz)

with a long 0.8s decay — the "wrong" dong.

💡 No audio files needed! This is a neat trick for vibe coders: you can create feedback sounds entirely in code. The Web Audio API is supported in all modern browsers. No loading MP3s, no file management, just ~30 lines of code per sound.

The Secret Sauce: Framer Motion

Fandom Trivia uses Framer Motion (now part of Motion) for nearly every animation. Instead of writing CSS keyframes manually, the app uses React-friendly animation components.

The three most common patterns:

🚪

Page Transitions

Every view wraps in <motion.div initial={{ opacity: 0 }} animate={{ opacity: 1 }} exit={{ opacity: 0 }}> — pages fade in when they mount and fade out when they leave.

🖱️

Hover Effects

Universe cards use whileHover={{ scale: 1.03, y: -4 }} and whileTap={{ scale: 0.98 }} — cards subtly lift when you hover and press down when clicked.

🎯

Spring Animations

Results screen uses transition={{ type: 'spring', stiffness: 200, damping: 20 }} — a bouncy, natural-feeling scale-in for the quiz complete card.

Badges: The Reward System

When a user finishes a quiz, the app runs evaluateBadges(). This function checks three things to award special titles:

CODE 

const evaluateBadges = (
  quizId: string,
  correctCount: number,
  total: number
) => {
  const badges = [];
  if (correctCount === total)
    badges.push({ type: 'perfect', });
  const universeBadge = BADGES
    .find(b => quizId
      .toLowerCase()
      .includes(b.universe));
  if (universeBadge)
    badges.push(universeBadge);
  return badges;
};
PLAIN ENGLISH

A function that checks if the player earned special rewards.

Which quiz was taken.

How many they got right.

Total questions.

If they got EVERY question right → plus a "Perfect Score!" badge.

No duplicates allowed.

Look up the badge config for this universe...

(e.g., "Twilight" → vampire badge).

If a matching badge exists → add it.

Return the collection of earned badges.

These display on the quiz complete screen.

🎮 Game design pattern: Badges are a simple but powerful motivation tool. This pattern — check conditions, award a title, display it — is the same logic used by Xbox Achievements, Duolingo streaks, and GitHub contribution badges. As a vibe coder, you can copy this pattern into any app that needs a reward system.
