Data & Users

How the app remembers you and your scores.

The Invisible Backend

Fandom Trivia doesn't run its own server. Instead, it uses Supabase — a cloud platform that provides both a database and a login system out of the box. Think of it like renting a storage unit: you don't build the building, you just get a key.

🧠 Why this matters: Most modern apps don't write their own server code — they use services like Supabase, Firebase, or Clerk. Understanding this pattern helps you evaluate costs and capabilities: "Can we do X with Supabase, or do we need a custom backend?"

The connection to Supabase is just six lines of code:

CODE 

import { createClient } from
  '@supabase/supabase-js';

const supabaseUrl = import.meta.env
  .VITE_SUPABASE_URL;
const supabaseAnonKey = import.meta.env
  .VITE_SUPABASE_ANON_KEY;

export const supabase = createClient(
  supabaseUrl, supabaseAnonKey
);
PLAIN ENGLISH

Import the Supabase client library — the toolbox for talking to our database.

Read the database URL from environment variables

(set in .env.local, never committed to GitHub).

Read the anonymous API key — it's not secret,

but it's locked down to only allow certain operations.

Create the client and export it

so any component can import and use it.

The Bouncer at the Door

When you click "Login," the app calls Supabase's OAuth flow — it redirects you to a Google login page, then brings you back with a verified identity. No passwords are stored in the app.

CODE 

const handleLogin = async () => {
  const { error } = await supabase.auth
    .signInWithOAuth({
      provider: 'google',
      options: {
        redirectTo: window.location.origin,
      },
    });
};
PLAIN ENGLISH

An async function — it takes time because we're talking to Google's servers.

Call Supabase's authentication service...

and ask to sign in using OAuth (delegated login).

Use Google as the identity provider.

After successful login,

come back to the same page we started on.

End of configuration block.

End of the function call.

💡 Key insight: The app never sees your password. Google handles the authentication, then sends a token to Supabase saying "this user is who they say they are." This is called OAuth — and it's the same system used by "Login with Google" on essentially every major website.

How Scores Survive Page Reloads

When you finish a quiz, your score is saved to a Supabase table called scores. This is why the leaderboard doesn't reset when you refresh — it's stored in the cloud.

💾

Saving

After quiz complete, MCQuizView calls .insert() to add a row to the scores table with your user ID, quiz name, score, and total.

📡

Reading

RankingsView calls .select().order('score', 'desc') to fetch top scores, grouped by quiz. Only the top 10 per quiz are shown.

👤

User Identity

Each score is linked to a user_id. When fetched, it joins with the profiles table to get the username.

Fetching the Leaderboard

Here's the actual code from RankingsView that pulls scores from the database.

CODE 

const { data, error } = await supabase
  .from('scores')
  .select(`
    id, score, total, quiz_id,
    created_at, user_id,
    profiles(username)
  `)
  .order('score', { ascending: false });
PLAIN ENGLISH

Ask Supabase for data (await = wait for the network).

From the "scores" table (like a spreadsheet tab).

Request these columns (fields):

unique row ID, numeric score, max total, quiz name.

timestamp, user ID (foreign key)...

and the username from the linked profiles table.

Close the column selection.

Sort by score, highest first (descending).

The #1 scorer is the first row returned.
