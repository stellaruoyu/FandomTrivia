Week 3 — Supabase Auth Migration

February 24, 2026 · The Big Migration · Supabase OAuth PostgreSQL RLS

This was a pivot week. The custom Express + SQLite + session auth was replaced wholesale with Supabase Auth — Google OAuth, PostgreSQL, Row Level Security. Everything changed: how users sign in, how scores are stored, and how data is secured.

Key Concepts

Why Supabase?

The original Express/SQLite backend worked for local development but had problems for production:

  • Session management needed server-side state — not ideal for an SPA
  • SQLite doesn't scale — no concurrent write support
  • Auth flows had to be built from scratch (password hashing, JWT, refresh tokens)
  • No RLS — all data security was at the application layer

Supabase solved all of these: hosted PostgreSQL, built-in auth with OAuth, realtime, and row-level security.

Supabase Client Setup

The core of the migration was src/supabaseClient.ts:

import { createClient } from '@supabase/supabase-js';

const supabaseUrl = import.meta.env.VITE_SUPABASE_URL;
const supabaseKey = import.meta.env.VITE_SUPABASE_ANON_KEY;

export const supabase = createClient(supabaseUrl, supabaseKey);

The import.meta.env pattern is Vite's way of handling environment variables. The Supabase URL and anon key are public — they're safe to expose because RLS policies prevent unauthorized access.

Google OAuth Flow

Sign-in became a single function call:

const { data, error } = await supabase.auth.signInWithOAuth({
  provider: 'google',
});

Supabase handles the entire OAuth dance: redirect to Google consent screen, callback handling, JWT token generation, and session management. The React app checks auth state via:

supabase.auth.onAuthStateChange((event, session) => {
  if (session) setUser(session.user);
  else setUser(null);
});

Profiles Table & RLS

User profiles were stored in a profiles table linked to auth.users via a DB trigger:

-- Trigger: auto-create profile on signup
create function public.handle_new_user()
returns trigger as $$
begin
  insert into public.profiles (id, username)
  values (new.id, new.email);
  return new;
end;
$$ language plpgsql security definer;

create trigger on_auth_user_created
  after insert on auth.users
  for each row execute function public.handle_new_user();

RLS policies ensured users could only access their own data:

create policy "Users can view own profile"
  on profiles for select
  using (auth.uid() = id);

⚠️ Lessons learned: When migrating from custom auth to Supabase, the biggest change isn't the code — it's the mental model. You go from "my server validates the session" to "the database enforces who can see what." RLS takes some getting used to, but once it clicks, it's more secure and less code.

Reflection

Week 3 was a complete auth overhaul. The switch to Supabase was motivated by production readiness: Google OAuth is more secure than a custom password flow, PostgreSQL handles concurrent users, and RLS provides database-level security. This migration was a net reduction in code (no more server routes for auth) and a net increase in security.
